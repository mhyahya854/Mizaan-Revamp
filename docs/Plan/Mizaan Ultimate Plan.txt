====================================================================
0. PURPOSE OF THIS FILE
====================================================================

This file is the single master plan for Mizaan.

It defines:
- what Mizaan is
- what Mizaan is not
- how the app should work from first launch to lifetime use
- how data should be stored
- how the portable USB/local vault model works
- how notes, documents, projects, people, finance, calendar, trackers, tasks, themes, graph, templates, backup, restore, and search should work
- how the current web prototype should evolve into a Windows desktop app
- how Android/iOS should be approached later
- what feature sets should be merged, improved, or removed
- what Codex must understand before writing code in future tasks

This is not a casual idea list.
This is the product direction.

Mizaan must be built as a lifetime app: boring where it must be boring, powerful where it matters, and always designed so the user’s data survives the app.

====================================================================
1. THE CORE PRODUCT LAW
====================================================================

Mizaan is a 100% local-first personal life operating system.

The user owns the data.
The vault owns the data.
The app edits the data.

The app must never become the only place where data exists.

Golden rule:

    If Mizaan dies, the data must survive.

That means:
- no forced cloud
- no forced login
- no forced account
- no Google dependency
- no Microsoft dependency
- no Notion dependency
- no server dependency
- no subscription dependency
- no hidden proprietary-only data trap

The app can become powerful, but the user must always be able to inspect, back up, copy, move, restore, and export the vault.

====================================================================
2. FINAL PRODUCT IDENTITY
====================================================================

Mizaan is:

    A portable local-first life operating system built around a user-selected vault folder.

It combines the best useful parts of:
- Notion-style pages, templates, and databases
- Obsidian-style local ownership, backlinks, graph, and markdown survival
- Logseq-style blocks, references, journals, and connected thinking
- Anytype-style objects and relations
- TickTick-style tasks, calendar, habits, reminders, focus, and review
- QOwnNotes-style durable local note ownership
- Tarteeb-style disciplined local file and document management

But Mizaan is not a clone.

Mizaan’s own identity:

    Notes + Documents + Projects + People + Finance + Calendar + Trackers + Tasks + Goals + Graph + Local Vault

Everything is local.
Everything is linkable.
Everything is recoverable.
Everything is organized inside a portable vault folder.

====================================================================
4. THE PORTABLE VAULT MODEL
====================================================================

The easiest way to solve the local/sync conflict:

    Mizaan stores and creates everything inside a portable vault folder.

The vault folder can live on:
- USB drive
- external SSD
- internal laptop folder
- portable hard drive
- SD card
- later, user-controlled NAS/local network folder
- later, user-chosen folder sync tool, but not app-managed cloud

The app itself is just the editor/manager.
The vault is the source of truth.

Typical flow:

1. User installs Mizaan on Windows.
2. User plugs in USB drive.
3. User opens Mizaan.
4. Mizaan asks: Create new vault or open existing vault.
5. User chooses USB folder, e.g. F:\Mizaan Vault.
6. Mizaan creates the full vault structure there.
7. User creates notes, imports documents, creates projects, links people, tracks finance, schedules calendar events, and builds graph links.
8. Mizaan writes everything into that vault.
9. User closes Mizaan.
10. User safely ejects USB.
11. User plugs USB into another laptop.
12. User opens Mizaan.
13. User opens the same vault.
14. All data is available.

This is not cloud sync.
This is portable vault continuity.

====================================================================
5. APP START TO END — FULL USER JOURNEY
====================================================================

--------------------------------------------------------------------
5.1 First launch
--------------------------------------------------------------------

User opens Mizaan for the first time.

Screen:
    Welcome to Mizaan
    Your private local life operating system.
    No account. No cloud. Your data lives in your vault.

Buttons:
- Create New Vault
- Open Existing Vault
- Restore from Backup
- Demo Mode

--------------------------------------------------------------------
5.2 Create new vault
--------------------------------------------------------------------

User clicks Create New Vault.

Mizaan asks:
- Vault name
- Vault location
- Storage mode

Default:
    Mizaan Vault

User chooses location:
- This computer
- USB/external drive
- Choose folder manually

Example:
    F:\Mizaan Vault

Mizaan runs safety checks:
- folder exists or can be created
- folder is writable
- temporary file can be written
- temporary file can be deleted
- enough storage is available
- vault identity file does not conflict with another vault
- if USB/external drive is detected, show portable vault notice
- database can be created
- backup folder can be created

Then Mizaan creates the vault.

--------------------------------------------------------------------
5.3 Open existing vault
--------------------------------------------------------------------

User clicks Open Existing Vault.

Mizaan asks for a folder.

Mizaan checks:
- does mizaan.vault.json exist?
- does mizaan.sqlite exist?
- is schema version supported?
- is vault locked?
- are attachments present?
- is database healthy?
- is backup status acceptable?

If valid:
    Open vault.

If missing files:
    Offer repair.

If locked:
    Offer read-only / force unlock / cancel.

If not a vault:
    Ask whether to create a new vault there.

--------------------------------------------------------------------
5.4 Restore from backup
--------------------------------------------------------------------

User chooses a backup ZIP.

Mizaan:
- validates backup manifest
- validates schema version
- validates database snapshot
- validates attachment list if available
- shows restore preview
- asks where to restore
- creates restored vault
- rebuilds indexes
- opens vault

Never restore blindly.

--------------------------------------------------------------------
5.5 Normal app opening
--------------------------------------------------------------------

On launch, Mizaan tries to open the last used vault.

If found:
- validate vault
- check lock
- open database
- run migration if needed
- check backup age
- rebuild damaged indexes if needed
- open Home

If missing:
    Last vault not found. Connect USB or choose another vault.

Options:
- Retry
- Open another vault
- Restore backup
- Demo Mode

--------------------------------------------------------------------
5.6 Normal app closing
--------------------------------------------------------------------

On close, Mizaan:
- flushes pending writes
- updates updatedAt and vault metadata
- optionally writes markdown mirrors
- writes close log
- removes vault lock
- warns if backup is overdue

If vault is on USB:
    Suggest safe eject after app closes.

8. LOCK FILE SYSTEM
====================================================================

When Mizaan opens a vault, it creates:

    .mizaan-lock

Example:

{
  "deviceId": "device_uuid",
  "deviceName": "Mohammad-PC",
  "openedAt": "2026-05-28T00:00:00Z",
  "appVersion": "0.1.0",
  "processId": "optional"
}

Purpose:
- prevent the same vault from being edited on two devices at once
- protect USB/external drive use
- help recover from bad shutdown

If lock exists:
- show warning
- allow read-only open
- allow force unlock only after confirmation
- show device name and last open time

Rules:
- app removes lock on clean close
- stale lock requires recovery prompt
- never silently overwrite lock

type: note
category: notes
created: 2026-05-28T00:00:00Z
updated: 2026-05-28T00:00:00Z
tags: [mizaan, vault, planning]
relations: []
---

Readable markdown content here.

Purpose:
- data survives app death
- external inspection possible
- easy export/import
- future migration easier

Rules:
- database is authoritative during app runtime
- mirror files are updated after debounce
- mirror regeneration must be possible
- deleted mirror should be repairable from database
- damaged database should be partially recoverable from mirrors

26. GRAPH SYSTEM
================

Mizaan needs a complete graph system inspired by Obsidian, Logseq, Anytype, mind-mapping tools, and manual diagram tools.

The graph system must support three different graph layers:

1. Global Graph

   * automatic graph of the whole vault
   * generated from real Mizaan data
   * used for vault-wide discovery

2. Automatic Local Graph

   * automatic graph around the current page/item
   * generated from backlinks, outgoing links, relations, attachments, tags, and nearby items
   * used to understand the current page context

3. Manual Local Graph / Page Canvas Graph

   * user-created editable graph
   * saved inside one page, document, project, person profile, or vault canvas
   * does not require graph nodes to be real Mizaan pages
   * supports editable nodes, arrows, edge labels, custom relationship types, family trees, mind maps, concept maps, and custom diagrams

Important rule:

```
Global Graph and Automatic Local Graph are generated by Mizaan.
Manual Local Graph is drawn by the user.
```

Phase 5 — Editable Simple Table and Basic Database Foundation - partially implemented — simple table blocks and basic database table pages are editable and provider-backed; formulas, rollups, advanced views, and the final database engine remain later.
Phase 6 — Native Windows/Tauri Readiness Boundary ❌ — blueprint exists, implementation not started.
Portable folder vault ❌ — real folder picker, lock file, and USB vault workflow are not implemented.
SQLite provider ❌ — not implemented.
Markdown mirrors ❌ — not implemented.

The graph system must not be fake UI.

26.1 CORE GRAPH CONCEPT
=======================

The graph is not a separate app.

The graph is a visual map of Mizaan’s data.

It does not replace pages.
It does not replace files.
It does not replace folders.
It does not replace databases.
It shows how everything is connected.

Mizaan graph data comes from:

* pages
* notes
* documents
* imported files
* projects
* people
* finance records
* calendar events
* tasks
* trackers
* goals
* tags
* dates
* attachments
* relations
* backlinks
* wiki links
* parent/subpage structures
* database relations
* manual graph blocks

Core rule:

```
The page is the real thing.
The graph shows how pages/things connect.
```

Example:

Page: Anatomy Cranial Nerves

* links to Page: Anatomy Exam
* links to File: Cranial Nerve Diagram.png
* tagged #anatomy
* mentions Date: 2026-06-23

Graph:

* Anatomy Cranial Nerves ── linked_to ── Anatomy Exam
* Anatomy Cranial Nerves ── attached_file ── Cranial Nerve Diagram.png
* Anatomy Cranial Nerves ── tagged ── anatomy
* Anatomy Cranial Nerves ── mentions_date ── 2026-06-23

=======================================

Mizaan uses this hierarchy:

Vault
Space

Properties

Example:

Mohammad's Mizaan Vault

├── Anatomy Cranial Nerves
├── Cranial Nerve Diagram

Definitions:

Vault:

* the full local folder/database system
* contains all Mizaan data

Space:

* top-level area such as Notes, Documents, Projects, People, Finance, Calendar, Tasks, Trackers, Goals

Page / Item:

* the main object the user opens and edits
* examples: note, document record, project page, person profile, task, event, goal

Block:

* content inside a page
* examples: paragraph, heading, checklist, table, image, file, graph block

Relation:

* connection between pages/items

Attachment:

* imported or linked file

Property:

* structured metadata such as status, date, tags, priority, amount, person, project

* calendar_task

---

## 26.4.1 Edge direction and arrows

Edges must support arrows.

Every edge can be:

1. Undirected
   Example:
   Person A ── married to ── Person B

2. Source to target
   Example:
   Parent ─────────▶ Child

3. Target to source
   Example:
   Child ◀──────── Parent

4. Bidirectional
   Example:
   Person A ◀──────▶ Person B

Edge direction options:

* undirected
* source_to_target
* target_to_source
* bidirectional

This is required for family trees, task dependencies, cause/effect diagrams, project maps, study diagrams, and manual concept maps.

For manual graphs, user can choose to always show labels.

## 26.4.3 Edge visual styles

* solid

* dotted

* warning
* custom

Mizaan detects this and creates an automatic graph edge.

* edge resolves to that page

If target page does not exist:

* create unresolved link node

* create page

* rename link

## 26.5.4 Attachments/files

User attaches/imports a file.

Anatomy Cranial Nerves ── attached_file ── cranial-nerve-diagram.png

If the file has a Document Page, edge should connect to the Document Page first.

## 26.5.5 Dates

User adds a date mention or calendar relation.

Example:

Graph edge:

Page ── mentions_date ── 2026-06-23

Date nodes can help show exam dates, deadlines, birthdays, and calendar planning.

---

## 26.5.6 People mentions

User mentions a person.

Example:

@Ahmed

Graph edge:

Current Page ── mentions_person ── Ahmed

If Ahmed has a Person Page, the mention links there.
If not, Mizaan may suggest creating a Person Page.

---

## 26.5.7 Parent/subpage relationship

If a page is nested under another page:

Parent Page
Subpage

Graph edge:

Parent Page ── parent_of ── Subpage

Direction:

Parent Page ─────────▶ Subpage

---

## 26.5.8 Database relations

If a database row has a relation property:

Task: Add SQLite Provider
Project: Mizaan Windows App

Graph edge:

Task: Add SQLite Provider ── belongs_to_project ── Mizaan Windows App

26.9 HOW THE GRAPH INDEX IS BUILT
=================================

The graph indexer runs when:

* page saved
* relation added
* relation removed
* file imported
* file removed
* tag added
* tag removed
* page renamed
* page deleted
* page restored
* attachment added
* attachment removed
* database relation changed
* full reindex requested
* migration completed
* backup restored

Graph indexing process:

1. Read item/page metadata.
2. Read blocks.
3. Parse links:

   * [[Page]]
   * #tag
   * @person
   * dates
   * file mentions
   * block references later
4. Read explicit relations.
5. Read attachments.
6. Read parent/subpage relationships.
7. Read database relation properties.
8. Generate graph_edges.
9. Update backlinks cache.
10. Update unresolved links.
11. Update tag graph.
12. Update graph cache.
13. Save graph health status.

Important rule:

```
The graph is not manually drawn every time.
It is generated from the user’s real Mizaan work.
```

26B. MANUAL LOCAL GRAPH / PAGE CANVAS GRAPH
===========================================

Mizaan needs a third graph type called Manual Local Graph, also called Page Canvas Graph.

This is different from the automatic Global Graph and automatic Local Graph.

Definitions:

Global Graph:

* automatic graph of the whole vault
* generated from pages, files, links, tags, relations, tasks, dates, and attachments
* used for discovery across the whole vault

Automatic Local Graph:

* automatic graph around the current page/item
* generated from backlinks, outgoing links, relations, attachments, tags, and nearby items
* used to understand the current page context

Manual Local Graph / Page Canvas Graph:

* user-created editable graph
* saved inside one page, document, project, person profile, or vault canvas
* does not require graph nodes to be real Mizaan pages
* nodes can be plain editable labels
* edges can be manually drawn and labeled
* edges can have arrows, direction, labels, and custom styles
* useful for family trees, mind maps, concept maps, study diagrams, project maps, relationship maps, and custom diagrams

Important rule:

```
Manual Local Graph is for human-made diagrams.
It does not need to follow the automatic vault graph.
```

## 26B.4.2 Edge direction and arrows

Every manual edge can have a direction.

Direction options:

1. Undirected
   Used when relationship is equal or mutual.

   Example:
   Person A ── married to ── Person B

2. Source to target
   Used when relationship flows from one node to another.

   Example:
   Parent ─────────▶ Child

3. Target to source
   Used when direction is reversed.

   Example:
   Child ◀──────── Parent

4. Bidirectional
   Used when both sides point to each other.

   Example:
   Person A ◀──────▶ Person B

This is required.

Manual graph must allow arrows from one node to another node.

Examples:

Father ── parent of ──▶ Child
Mother ── parent of ──▶ Child
Cause ── leads to ──▶ Effect
Task A ── blocks ──▶ Task B
Person A ◀── sibling of ──▶ Person B

40. WARNINGS
====================================================================

1. USB portable vault is not true sync.
It is portable storage. It works best when only one device edits the vault at a time.

2. Do not open the same vault on two computers at once.
This can cause conflicts or database corruption.

3. Use a lock file.
Mizaan must warn when a vault may already be open elsewhere.

4. Do not build forever on localStorage.
It is acceptable for prototype work, but not lifetime functionality.

5. Do not sync a live SQLite database through random cloud/network folders.
Use backup packages or app-controlled change packages later.

6. USB drives can fail.
Mizaan must encourage backups to another location.

7. Android can work later, but mobile storage is more restricted than Windows.
Start mobile as a companion, not full parity.

8. iOS is harder for this exact USB vault model.
Treat iOS as import/export companion first.

9. Do not build mobile before Windows vault architecture is stable.
Otherwise the storage system will be rewritten multiple times.

10. Do not add every Notion/TickTick feature blindly.
Merge useful feature categories, remove cloud/team/account/public/billing features, and keep Mizaan focused on lifetime local ownership.

11. Backup is not optional.
For a local app, backup is the safety net that cloud apps usually pretend to provide.

12. Data portability is the product promise.
If data becomes trapped, Mizaan has failed.

- load active reminders

- respect snoozed reminders
- mark missed reminders when the app was closed
- reschedule recurring reminders
- cancel reminders when source item is deleted
- update reminders when source item changes
- log notification delivery

Scheduler behavior:

- when app opens, check missed reminders
- while app is open, check scheduled reminders at safe intervals
- when app closes, persist scheduler state
- when user changes time/date/timezone, re-check reminder timing

--------------------------------------------------------------------

- allow user to snooze

- allow user to mark task/event as done if applicable

- keep reference to original source item
- appear in notification history

--------------------------------------------------------------------

- mark reminder as dismissed

- not delete the reminder record immediately
- keep history for audit/review

====================================================================
58. MIGRATION STRATEGY
====================================================================

Mizaan will evolve over time, so migrations must be safe.

Core rule:

    Never migrate without a backup.

--------------------------------------------------------------------
58.1 Migration rules
--------------------------------------------------------------------

Every migration must:

- have a version number
- have a name
- have a timestamp
- create pre-migration backup
- log what changed
- run inside a transaction where possible
- validate result
- fail safely
- restore backup if needed
- update schema version only after success

--------------------------------------------------------------------
58.2 Migration flow
--------------------------------------------------------------------

When app opens a vault:

1. read current schema version
2. compare with app-supported schema version
3. if schema is older, show migration required
4. create pre-migration backup
5. run migrations in order
6. validate database
7. rebuild indexes if needed
8. update schema version
9. open vault

If migration fails:

- stop opening vault
- show error
- restore pre-migration backup if possible
- write migration log
- offer manual repair

--------------------------------------------------------------------
58.3 Too-new vault
--------------------------------------------------------------------

If vault schema is newer than the app supports:

Do not open normally.

Show:

    This vault was created or updated by a newer version of Mizaan.
    Please update Mizaan before opening this vault.

Options:

- Open read-only if safe
- Cancel
- Locate backup

Never downgrade automatically.

--------------------------------------------------------------------
58.4 Migration logs
--------------------------------------------------------------------

Migration logs live in:

    logs/migration.log

Each log entry includes:

- migration id
- from version
- to version
- started_at
- completed_at
- success/failure
- error message
- backup path
